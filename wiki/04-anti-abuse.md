# 04. Анти-абуз

## MVP-защита

1. **Telegram initData HMAC.** Backend принимает только подписанные Telegram данные.
2. **Один прогноз на событие.** Ограничение на уровне БД: `UNIQUE(event_id, user_id)`.
3. **Закрытие после старта.** Если `starts_at` уже наступил, событие автоматически блокируется при попытке прогноза.
4. **Статусы событий.** Админ может закрыть событие вручную: `locked` или `settled`.
5. **Rate limit.** Простой in-memory лимит запросов по initData/IP.
6. **Блокировка пользователя.** Есть поле `is_blocked` для ручной блокировки.

## Что добавить перед публичным запуском

- Redis rate limit, чтобы лимит работал на нескольких инстансах.
- Логи подозрительных действий: частая смена прогнозов, много новых аккаунтов за минуту.
- Ограничение изменения прогноза за N минут до старта.
- Проверка членства в конкретном Telegram-канале/чате через Bot API, если прогнозы только для своей аудитории.
- Cloudflare Turnstile только для веб-открытий вне Telegram, если такие появятся.
- Админский журнал: кто внёс/изменил результат.

## Что не решает Telegram ID

Telegram ID не равен реальной личности. Один человек теоретически может иметь несколько аккаунтов. Для фан-прогнозов это приемлемо; для денежных операций — нет, там нужен KYC и юридическая модель.
